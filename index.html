Content-type对get和post的影响

发表于 2020-09-17 更新于 2020-09-22 分类于 HTTP

一. GET 请求

GET 请求不存在请求实体部分，键值对参数放置在 URL 尾部，浏览器把form数据转换成一个字串（name1=value1&name2=value2…），然后把这个字串追加到url后面，用?分割，加载这个新的url。因此get请求头不需要设置 Content-Type 字段

​ 非 ASCII 码会自动进行编码转换，例如发送请求：www.bilibili.com?hehe=你的我的

​ 值得一提的是，GET 参数的编码方式是无法人为干涉的，这导致了不同浏览器有不同的编码方式，因此最稳妥的方案是人工预编码，人工解码，从而禁止浏览器编码的干涉

二. POST 请求

Http Header里的Content-Type一般有这三种：

application/x-www-form-urlencoded(表单默认)

​ 最常见 POST 提交数据的方式。浏览器的原生 form 表单，如果不设置 enctype 属性，那么最终就会以 application/x-www-form-urlencoded 方式提交数据。

​ 数据被编码为名称/值对。这是标准的编码格式。会将表单内的数据转换拼接成 key-value 对（非 ASCII 码进行编码）。

multipart/form-data(一般用来上传文件)

​ 这是另一种非常常见的 POST 数据提交的方式。我们在使用表单上传文件时，必须让 form 的 enctyped 等于这个值。关于处理上传请求的实战后续我会写文章来补充。

application/json

​ application/json 关于这个 Content-Type 作为响应头大家应该并不会感到不陌生。实际上，现在越来越多的人把它作为请求头，用来告诉服务端消息主体是序列化后的 JSON 字符串。由于 JSON 规范的流行，除了低版本 IE 之外的各大浏览器都原生支持 JSON.stringify，服务端语言也都有处理 JSON 的函数，使用 JSON 不会遇上什么麻烦。

text/plain

​ 数据以纯文本形式(text/json/xml/html)进行编码，其中不含任何控件或格式字符。postman软件里标的是RAW。（中文不进行编码）

也就是说，字符串由一些由空格分隔的部分组成，你需要将这些部分逆序。
给定一个原字符串 A ：“dog loves pig” 和他的长度，请返回逆序后的字符串 “pig loves dog”。

思路

  1. 首先将句子中所有内容都进行逆序，dog loves pig -> gip sevol god
  2. 再将每个单词中的所有内容进行逆序， gip sevol god -> pig loves dog
